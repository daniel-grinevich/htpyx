Diving into our Infrustructure Choices
Subheader 1

The infrastructure built for this project comprises a Raspberry Pi, an Eversys espresso machine, and a custom web server. We chose the Raspberry Pi for its cost-effectiveness and versatility in controlling the espresso machine's operations and managing data. The custom web server supports real-time order processing and customer management. These choices were strategically made to keep operational costs low, enhancing our financial sustainability. Additionally, the modular nature of our setup allows for easy scalability with minimal investment, ensuring we can expand our services efficiently as demand grows.

The way all these components interact with the espresso machine is through serial communication and the Eversys API. The Eversys API is a dynamic link library, which was one of the reasons that led us to choose to code in C#. A significant advantage of using C# was leveraging the Xamarin framework, allowing us to compile natively for both iOS and Android. Our web framework is built with ASP.NET Core, complementing our other infrastructure choices and ensuring a seamless integration across all platforms.

How Everything Fits Together
Subheader 2

Now that we have a general understanding of the key components driving our automated espresso machine, let’s delve deeper into the specifics of each element. This will help us undeerstand how they all fit into the bigger puzzle.

The App Code

The app code serves as the primary interface for our users. It features a dynamic cart system that allows users to add, delete, and edit items in their order. Within this app, we define the various coffee options available, along with any customization options. After users finalize their selections and the order passes initial validation, it is then sent to our web server for processing. This streamlined interaction ensures a user-friendly experience while maintaining the accuracy and efficiency of the order system.

The web server

Our web server plays a critical role in managing the coffee ordering process. It constructs a coffee interface that not only builds orders but also maintains a state tracking system. This system monitors the order’s progress—from confirmation to whether the espresso machine, controlled by the Raspberry Pi, has begun making the coffee. Additionally, the server manages a JSON database where coffee options are stored and updated. Once an order is confirmed, it is dispatched to the Raspberry Pi for execution.

Raspberry Pi

The Raspberry Pi serves as a crucial control hub in our coffee-making process. Its primary role is to monitor for QR code scans that trigger the brewing process. Upon detecting a QR scan, the Raspberry Pi sends a command via the serial port to the Eversys espresso machine, initiating the coffee preparation. Concurrently, it controls a syrup pump, integrating any specified syrups into the coffee as per the order details. Once the coffee is ready, the Raspberry Pi sends a confirmation signal back to the server, indicating that the order has been completed.

Solution #1

The challenge with the caramel syrup pump was a messy one. It continually dispensed syrup without stopping, leading to both wastage and a messy work environment. To address this, we designed a state machine that was integrated into our system's web server. The state machine monitored the different stages of the coffee-making process, ensuring that syrup was dispensed only at the correct step. Not only did this solution fix the overflow issue, but it also introduced a valuable feature: the ability to lock an espresso machine to a single order, effectively preventing the possibility of double-scanning a coffee.

Solution #2

Our next challenge emerged during the integration of the QR code scanner. We had initially coded a Python script to interact with the scanner; however, it failed to function as expected. The issue stemmed from our lack of knowledge about the appropriate scanner model to purchase. The model we acquired sent responses in the form of key presses, which our system could not interpret correctly. To resolve this, we had to modify and rewrite our scanner code, enabling it to translate these key presses into meaningful commands within our order processing system.

Solution #3

Although we never fully implemented this third solution, the idea was to overhaul the communication protocol by recoding the existing dynamic link library (DLL) in Python. The goal was to utilize byte manipulation to simplify the API, making it more robust and easier to maintain. This would have allowed us to have more granular control over the hardware interactions, potentially leading to even more precise control and diagnostics capabilities across the system.
